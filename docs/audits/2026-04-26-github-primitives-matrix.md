# GitHub Primitive Matrix for Minis Ingestion Planning

Date: 2026-04-26  
Scope: Current `backend/app/ingestion/github.py`, `backend/app/plugins/sources/github.py`, and local-clone primitives in `backend/app/explorer/`.

## Matrix

| Primitive | Why it matters for mini fidelity | API source (REST/GraphQL endpoint family) | Bulkability | Reconstructable from local clone? | Current status in repo | Recommended ingestion owner path | Priority | Risk / permission notes (org-data constraints) |
|---|---|---|---|---|---|---|---|---|
| Profile (bio, location, company, hireable, social graph counts) | Core identity framing and audience/context priors for voice + decision lens | REST `GET /users/{username}` (optionally GraphQL `user`) | High | No | Partial | API bulk | P1 | Public profile is low risk; private profile fields require user token scopes and explicit consent logging. |
| Repos metadata (name, topics, primary language, pushed_at, fork/archived, stars) | Shows domains, long-term interests, maintenance behavior, and project context switching | GraphQL `user.repositories` (fallback REST `GET /users/{username}/repos`) | High | Partial | Partial | API bulk | P0 | Org/private repos require org-approved token + SSO; today only owner repos are targeted. |
| Repo language breakdown | Strong signal for technical breadth vs depth and stack shifts over time | GraphQL `repository.languages` (fallback REST `GET /repos/{owner}/{repo}/languages`) | High | Partial | Ingested (summary + metadata attach) | API bulk | P1 | Private/org repos need read scope; language stats can expose private stack choices. |
| Stars (repositories the subject starred) | Preference/taste graph and external learning signal | REST `GET /users/{username}/starred` | High | No | Ingested | API bulk | P1 | Star lists are public by default, but enterprise may treat them as behavioral data; document retention policy. |
| Watches / subscriptions | Attention graph is often a better recency signal than stars | REST `GET /users/{username}/subscriptions` or GraphQL watchers/subscriptions fields | High | No | Missing | API bulk | P0 | Requires additional scopes for private subscriptions; org policy may forbid collecting watch/subscription behavior. |
| PRs authored | Primary record of authored tradeoffs, framing, and communication style | REST Search `GET /search/issues?q=author:{u} type:pr` | Medium | Partial | Partial | API bulk | P0 | Search caps/rate limits can silently truncate; private/org PRs require appropriate scopes. |
| PRs commented/reviewed (not authored) | Highest-value signal for decision framework under critique | REST Search `GET /search/issues?q=commenter:{u} type:pr` + GraphQL authored-review contributions | Medium | Partial | Partial | API bulk | P0 | Incomplete without full PR universe access; org/private review history needs explicit opt-in. |
| PR review state events (approve/request changes/comment) | Captures decision outcomes, reversals, and blocking thresholds | REST `GET /repos/{owner}/{repo}/pulls/{number}/reviews` | Medium | Partial | Ingested | API bulk | P0 | Review visibility follows repo permissions; private enterprise repos add confidentiality constraints. |
| Inline review comments (file/line/diff-hunk anchored) | Most precise evidence of what the person flags and how they coach | REST `GET /repos/{owner}/{repo}/pulls/{number}/comments` | Medium | Partial | Ingested | API bulk | P0 | Sensitive if private repos are included; keep provenance + access class intact. |
| PR review threads (conversation chains) | Shows negotiation style, escalation, and conflict-repair patterns | Derived from PR review comment REST stream grouped by `in_reply_to_id` | Medium | Partial | Ingested | API bulk | P1 | Thread completeness depends on caps; org repos need same comment read permissions. |
| Issue comments (across issues/PRs) | Broader collaboration voice outside strict code review contexts | REST `GET /repos/{owner}/{repo}/issues/{number}/comments` (plus user events) | Medium | No | Partial | API bulk | P0 | Current coverage is mostly PR-linked; full issue corpus needs explicit issue-search expansion. |
| Issue threads (conversation snapshots) | Audience modulation and longitudinal problem-solving behavior | Derived from issue-comment fetches | Medium | No | Partial | API bulk | P1 | Thread snapshots include other participants; org policy may require participant-data minimization. |
| Non-PR issues (authored/commented issue bodies + lifecycle) | Critical missing surface for decision behavior outside PR flow | REST Search `GET /search/issues?q=author:{u} type:issue` and `commenter:{u} type:issue` | High | No | Missing | API bulk | P0 | Often includes internal incident/process context in org repos; enforce opt-in and scope control. |
| Timeline events (opened/closed/reopened/labeled/review-requested, cross refs) | Required for sequence-level framework inference and self-correction trajectories | REST `GET /repos/{owner}/{repo}/issues/{number}/timeline` (and PR timeline variants) | Medium | No | Missing | API bulk | P0 | Timeline may expose org structure/workflow internals; requires stricter enterprise permission model. |
| Reactions (PR, issue, review, comment emoji) | Lightweight approval/disapproval and social calibration signal | REST reactions endpoints (`/reactions` families on issues/comments/reviews) | Medium | No | Missing | API bulk | P1 | Reaction streams include third-party users; retention and user-minimization should be explicit. |
| Discussions (repo/org discussions + comments) | High-signal long-form design reasoning and public teaching style | GraphQL `repository.discussions` / `organization.discussions` families | Medium | No | Missing | API bulk | P0 | Org discussions often private and policy-sensitive; must be behind explicit org-data opt-in flag. |
| Commit comments | Captures line-level critique outside PR flow (important in some teams) | REST `GET /repos/{owner}/{repo}/comments` + commit-comment routes | High | No | Missing | API bulk | P1 | Private/org commit comments can leak internal context; scope + retention controls required. |
| Gists | Personal snippets, experiments, and explanatory style | REST `GET /users/{username}/gists` (+ raw file URLs) | High | No | Ingested | API bulk | P2 | Usually public but may include secrets accidentally; keep secret-scanning in downstream pipeline. |
| Gist comments | Peer feedback + explanatory interaction around snippets | REST gist comments families (`/gists/{gist_id}/comments`) | Medium | No | Missing | API bulk | P2 | Third-party commenter data; add participant-data handling guardrails. |
| Releases (release objects + notes) | Product narrative, change communication quality, and semantic version discipline | REST `GET /repos/{owner}/{repo}/releases` (or GraphQL release fields) | High | Partial | Missing | API bulk (with local fallback for changelog files) | P1 | Private release notes are org-sensitive; require explicit permission tiering. |
| Milestones + labels metadata | Encodes planning taxonomy and severity/priority language | REST `GET /repos/{owner}/{repo}/milestones` + `GET /repos/{owner}/{repo}/labels`; PR/issue attached labels | High | No | Missing | API bulk | P1 | Org process metadata may be considered internal governance data; gate behind org-data control. |
| Local-clone commit history (full, uncapped) | Avoids API truncation and provides complete evolution signal | Local git (`git log`, repo clone manager) | High | Yes | Partial | Local git mining | P0 | Requires successful clone + token for private repos; current fan-out depends on repo selection wiring. |
| Local-clone diff hunks (full patch fidelity) | Preserves complete code-change rationale beyond API truncation limits | Local git (`git show`, file diffs) | High | Yes | Partial | Local git mining (API detail only as fallback) | P0 | Large diffs can be expensive; enforce size budgets and per-repo bounds. |
| Local-clone file evolution (path-level history, rename-aware) | Enables longitudinal framework extraction at file/module level | Local git (`git log --follow -- <path>`) | Medium | Yes | Partial | Local git mining | P1 | Depends on robust path traversal protections (already present) and clone coverage of target repos. |

## Recommended sequencing

1. P0 now: watches/subscriptions, non-PR issues, timeline events, discussions, and local-git-first for commit/diff primitives.
2. P1 next: reactions, commit comments, releases, milestones/labels, and full issue-comment coverage.
3. Keep API as source-of-truth for social/collaboration primitives; use local git mining as source-of-truth for code-evolution primitives.
4. Add explicit org-data opt-in gating before enabling org/private discussions, teams, or private repo expansion.
5. Persist run-level completeness telemetry (caps hit, skipped counts, scope denied) so matrix status can be measured continuously.
